**File:** `fayed-mobile/app/_layout.tsx`

```
- Providers: SafeAreaProvider → QueryClientProvider → ThemeProvider → AuthProvider → NavigationHistoryProvider

```

### 2. (auth) Layout — `app/(auth)/_layout.tsx`
**File:** `fayed-mobile/app/(auth)/_layout.tsx`

```
- No auth state check whatsoever
- No role verification
- Simply renders <Stack screenOptions={{ headerShown: false }} />
- Auth routes are public; no redirect logic present
- Unauthenticated users can freely access all (auth) routes
```

### 3. (patient) Layout — `app/(patient)/_layout.tsx`
**File:** `fayed-mobile/app/(patient)/_layout.tsx`

```
- No auth state check whatsoever
- No role verification
- Renders <Tabs> with patient-facing tabs (home, sessions, notifications, profile)
- Uses href={null} to hide nested stack routes from tab bar
- Role protection is handled exclusively by AuthProvider's useEffect segments guard
- This layout does NOT prevent a practitioner from reaching these routes
```

### 4. (practitioner) Layout — `app/(practitioner)/_layout.tsx`
**File:** `fayed-mobile/app/(practitioner)/_layout.tsx`

```
- No explicit auth check (auth state read via useAuth() but no redirect)
- Role verified via useAuth().role
- Checks practitionerStatus === "APPROVED" to show/hide primary tabs
- Tab href logic:
    - index, sessions/index, availability/index, more: href=isApprovedPractitioner ? undefined : null
      (shown only when APPROVED)
    - application-status: href=isApprovedPractitioner ? null : undefined
      (hidden when APPROVED, visible when not APPROVED)
- Hidden routes (href={null}) are NOT protected by the layout; they are accessible if
  a practitioner navigates directly via router.push()
- Role protection is handled by AuthProvider's segments guard (same as patient)
```

### 5. AuthProvider Segments Guard (in `src/providers/AuthProvider.tsx`)
**File:** `fayed-mobile/src/providers/AuthProvider.tsx`, lines 408-501

```
- useEffect on [isBootstrapping, router, segments, session]
- Uses useSegments() to read first route segment
- If no session and not in (auth) group: router.replace("/(auth)")
- If patient session and in (auth) or (practitioner): router.replace("/(patient)")
- If practitioner session and in (auth) or (patient): router.replace("/(practitioner)")
- If practitioner session with practitionerStatus !== "APPROVED" and NOT on application-status:
    router.replace("/(practitioner)/application-status")
- If practitioner APPROVED and on application-status: router.replace("/(practitioner)")
- Race condition risk: redirect fires in useEffect after route renders; brief exposure window
```

## Token Storage Findings

### Token Storage Architecture
**Files:**
- `fayed-mobile/src/features/auth/secure-token-storage.ts`
- `fayed-mobile/src/features/auth/storage.ts`

### Token Storage Details

**Native (iOS/Android):**
- Access token: `SecureStore` key `fayed.mobile.auth.tokens.access.v1`
- Refresh token: `SecureStore` key `fayed.mobile.auth.tokens.refresh.v1`
- Access expiry: `SecureStore` key `fayed.mobile.auth.tokens.access.expiresAt.v1`
- Refresh expiry: `SecureStore` key `fayed.mobile.auth.tokens.refresh.expiresAt.v1`
- Session metadata (role, user): `AsyncStorage` key `fayed.mobile.auth.session.v2`
- **Tokens properly stored in SecureStore on native — NO plain AsyncStorage for tokens on native**

**Web:**
- Tokens stored in `AsyncStorage` with the same keys as native SecureStore
- **AUDIT-072 / P1 ISSUE:** Web uses AsyncStorage for tokens (no Keychain on web — acknowledged in comments at line 30-34 of secure-token-storage.ts)
- Comment explicitly states: "Web/Expo web: AsyncStorage (browser-compatible, no Keychain on web)"

### Refresh Token Logic
**File:** `fayed-mobile/src/providers/AuthProvider.tsx`, lines 408-449

```
- configureApiAuthSessionHandlers registers:
    refreshAccessToken: calls patientRefresh or practitionerRefresh with stored refreshToken
    onAuthFailure: clears session, redirects to /(auth)
- Axios interceptor at api.ts lines 138-175 handles 401 by:
    1. Retrying the request once after refresh
    2. If refresh fails (null returned), calls onAuthFailure → clearStoredAuthSession + router.replace("/(auth)")
- Refresh failure properly clears tokens via clearAuthenticatedState() → clearStoredAuthSession()
- No token left uncleared on refresh failure — fail-closed behavior
```

### Logout Clearing
**File:** `fayed-mobile/src/providers/AuthProvider.tsx`, lines 550-567

```
- signOut:
    1. Calls revokeCurrentPushRegistration()
    2. Calls patientLogout or practitionerLogout API (refresh token sent)
    3. Errors ignored (transport errors don't block local clear)
    4. clearAuthenticatedState() always called in finally block
    5. router.replace("/(auth)")
- clearStoredAuthSession() clears both SecureStore tokens AND AsyncStorage session metadata
- Properly clears ALL token artifacts
```

### Token Console Logging
**Grep result:** No matches for `console.*[Tt]oken` or `console.*[Aa]ccess` or `console.*[Rr]efresh`
**No evidence of token values being logged to console**

### Device ID
**File:** `fayed-mobile/src/features/auth/storage.ts`, lines 15, 154-163
- Device ID stored in `AsyncStorage` key `fayed.mobile.device.id.v1`
- Created as `device_${Date.now()}_${Math.random().toString(36).slice(2, 12)}`
- Not a security issue (device identifier, not a secret)

### Deep Link Tokens in URL
**No evidence found** of token values being passed in deep link URLs.
Deep link routing uses `extractNotificationHref(data)` from notification data payloads, not URL query params.

## Route Protection Findings

### Role Segregation
- **(auth) group**: Completely public. No guard prevents access.
- **(patient) group**: AuthProvider segments guard (useEffect on segments). Patient role required.
- **(practitioner) group**: AuthProvider segments guard. Practitioner role required.
- **No mixing**: Patient routes are NOT under (practitioner) group and vice versa. Strict file-system separation.

### Layout-Level vs. Route-Level Protection
- **Layout-level:** (patient) and (practitioner) layouts render tabs with no auth checks
- **Route-level:** AuthProvider's segments-based useEffect is the ONLY auth gate
- **Hidden routes (href={null}):** These are NOT protected by the layout; they are accessible if navigated to directly. However, the AuthProvider segments guard would redirect an unauthorized user before they could access them.

### Race Condition Window
- The segments guard fires in a `useEffect` after the component mounts
- A brief window exists where an unauthorized user could see route content before redirect
- **This is standard React Router behavior but worth noting for high-security screens**

### Patient Layout Cannot Access Practitioner Routes
- File-system separation: no patient routes inside `(practitioner)` folder
- AuthProvider redirects practitioner trying to access `(patient)` routes
- **Confirmed: No route group mixing**

### Practitioner Layout Cannot Access Patient Routes
- File-system separation: no practitioner routes inside `(patient)` folder
- AuthProvider redirects patient trying to access `(practitioner)` routes
- **Confirmed: No route group mixing**

## Practitioner OTP/Account State Findings

### OTP Verification Flow
**File:** `fayed-mobile/app/(auth)/signin/practitioner.tsx`, lines 52-122

```
- Practitioner login starts with startPractitionerLogin() (AuthProvider.tsx line 530)
- Backend returns either:
    1. AuthSuccessResponse (full session) — dev mode compatibility
    2. OtpChallengeResponse (requires OTP) — production flow
- If OtpChallengeResponse: UI shows OTP input screen
- verifyPractitionerOtp() called with challengeId + code
- On success: consumeAuthSuccess → session persisted → redirect based on role
- OTP challenge has expiresAt field — UI displays expiry time
```

### Account Approval Gate
**File:** `fayed-mobile/src/providers/AuthProvider.tsx`, lines 486-500

```
- After login, if session.role === "practitioner":
    - If practitionerStatus !== "APPROVED" AND NOT on application-status route:
        → router.replace("/(practitioner)/application-status")
    - If practitionerStatus === "APPROVED" AND on application-status route:
        → router.replace("/(practitioner)")
- application-status is the ONLY dedicated pending-approval screen
```

### Screens Accessible Before Account Approval
**File:** `fayed-mobile/app/(practitioner)/application-status.tsx`

```
- practitioner/application-status.tsx: accessible to non-approved practitioners
  (not redirected because it's the application-status route itself)
- practitioner/onboarding.tsx: hidden route (href={null} in layout) — accessible only
  if navigated to directly. Would not be accessible via tabs but can be reached via
  router.push() by a non-approved practitioner
- No other operational screens are accessible to non-approved practitioners
  (tabs hidden, other routes blocked by AuthProvider redirect)
```

### Practitioner Presence Heartbeat
**File:** `fayed-mobile/app/(practitioner)/_layout.tsx`, lines 31-33

```
- usePractitionerPresenceHeartbeat called with condition:
    role === "practitioner" && isApprovedPractitioner
- Heartbeat is NOT started for non-approved practitioners

```

### No Per-Screen Auth Middleware
- There is no expo-router middleware or per-screen auth wrapper
- All auth is centralized in AuthProvider's segments-based useEffect
- This means every protected screen relies on the same single guard

## Open Questions

1. **What happens if a practitioner's OTP expires during the login flow?** The UI shows `challenge.expiresAt` but there's no visible handling for expiry. If the user submits after expiry, the API would presumably return an error.

2. **What is the backend behavior if a practitioner tries to access operational routes without being APPROVED?** The mobile guard is client-side only — if the backend also enforces this, the mobile guard is defense-in-depth.

3. **Is there any server-side validation that the practitionerStatus in the session matches the database state?** If a practitioner is approved in-session but later revoked, does the app detect this?

4. **What happens if the `restoreSession()` bootstrap in AuthProvider fails partially?** E.g., getStoredAuthSession succeeds but getAuthMe fails and refresh also fails — the user is logged out. This is correct behavior but worth confirming UX is graceful.

5. **The `(practitioner)/onboarding.tsx` is a hidden route. How is it navigated to?** The only reference found is in `(practitioner)/onboarding.tsx` itself (line 554) where it navigates to `/(practitioner)/account`. Direct navigation to `/onboarding` is possible but would be caught by the approval gate redirect.

## Findings to Register (AUDIT-105 onwards)

---

### AUDIT-105
**Title:** No loading.tsx or error.tsx files exist in any route folder
**Severity:** Low
**Evidence:** Glob pattern `**/loading.tsx` and `**/error.tsx` in `fayed-mobile/app/` returned no matches
**Module:** Route Infrastructure
**Risk:** Loading and error states for individual routes are not handled via Expo Router's file-based error/loading conventions. All loading states are handled inline in components, which is inconsistent with Expo Router best practices and provides no route-level error boundary.

---

### AUDIT-106
**Title:** (auth) layout has no authentication check — relies entirely on downstream guard
**Severity:** Low
**Evidence:** `fayed-mobile/app/(auth)/_layout.tsx` is a bare `Stack` with no auth check. Any user (including authenticated patients/practitioners) can navigate to `(auth)` routes freely. The redirect back to the appropriate group is handled by `AuthProvider.tsx` useEffect.
**Module:** Auth Layout
**Risk:** While the downstream guard works, unauthenticated users can briefly render auth screens while the redirect fires. This is a defense-in-depth gap — the auth layout should also redirect authenticated users away from auth routes.

---

### AUDIT-107
**Title:** Hidden practitioner routes (href={null}) are not protected at the layout level
**Severity:** Low
**Evidence:** `fayed-mobile/app/(practitioner)/_layout.tsx` lines 119-189 list many routes with `href={null}` (hidden from tab bar). These routes (e.g., finance, messages, care-chat) are accessible if navigated to directly via `router.push()`, because the layout has no per-route auth check. The AuthProvider segments guard provides protection, but there is no layout-level enforcement.
**Module:** Route Protection
**Risk:** If the AuthProvider segments guard has a race condition or is bypassed, hidden routes could be accessed directly. Layout-level route protection would provide defense-in-depth.

---

### AUDIT-108
**Title:** Web platform stores auth tokens in plain AsyncStorage
**Severity:** Medium
**Evidence:** `fayed-mobile/src/features/auth/secure-token-storage.ts` lines 36-53 and 73-81. For `Platform.OS === "web"`, tokens are stored in `AsyncStorage` (browser localStorage), not a secure enclave. The code explicitly acknowledges this: "Web/Expo web: AsyncStorage (browser-compatible, no Keychain on web)."
**Module:** Token Storage
**Risk:** On web, tokens are stored in plain browser storage and could be read by JavaScript running on the same origin. This is a known limitation (no Keychain equivalent on web), but should be documented and, if possible, mitigated with additional measures such as httpOnly cookie fallback for web.

---

### AUDIT-109
**Title:** Practitioner OTP login has no client-side expiry enforcement
**Severity:** Low
**Evidence:** `fayed-mobile/app/(auth)/signin/practitioner.tsx` displays `challenge.expiresAt` in the UI (line 265-268) but the `submitOtp()` function (lines 104-121) does not check if the challenge has expired before sending the OTP code to the backend. The API presumably rejects expired OTPs, but the UX does not prevent the submission.
**Module:** Practitioner Auth
**Risk:** User submits an OTP after the challenge has expired without immediate client-side feedback, leading to a confusing error before they can request a new challenge.

---

### AUDIT-110
**Title:** All route protection is centralized in a single AuthProvider useEffect with a race condition window
**Severity:** Low
**Evidence:** `fayed-mobile/src/providers/AuthProvider.tsx` lines 408-501. The segments-based redirect runs in a `useEffect`. On initial load, the route component renders before the redirect fires. No per-route auth wrapper exists.
**Module:** Route Protection
**Risk:** For a brief moment after app launch and before the redirect fires, a user could see content of a route they are not authorized for. This is especially relevant if the user refreshes the app on a protected route.

---

### AUDIT-111
**Title:** (practitioner)/onboarding.tsx is accessible as a hidden route via direct navigation
**Severity:** Low
**Evidence:** `fayed-mobile/app/(practitioner)/_layout.tsx` line 135-137: `onboarding` is listed with `href={null}`. It is hidden from the tab bar but has no route guard preventing direct navigation. The AuthProvider approval gate at `AuthProvider.tsx` lines 486-500 redirects to `/application-status` for non-approved practitioners, but only if they are NOT already on the application-status route. A direct navigation to `/onboarding` would render the full onboarding workspace for a non-approved practitioner.
**Module:** Practitioner Route Protection
**Risk:** A non-approved practitioner could access the full onboarding workspace directly via `router.push('/(practitioner)/onboarding')`. While the data is likely filtered by the backend based on the practitioner's actual status, the UI should enforce the same restrictions as the tab bar.

---

### AUDIT-112
**Title:** Device ID is generated using Math.random() which is not cryptographically secure
**Severity:** Low
**Evidence:** `fayed-mobile/src/features/auth/storage.ts` line 18: `Math.random()` is used to generate device IDs: `device_${Date.now()}_${Math.random().toString(36).slice(2, 12)}`
**Module:** Token Storage
**Risk:** `Math.random()` is not cryptographically random. While device IDs are not secrets, a more secure random generator (e.g., `crypto.getRandomValues()` or `expo-crypto`) should be used to generate device identifiers to prevent potential device ID collision or prediction.

---

### AUDIT-113
**Title:** No explicit handling for PENDING_REVIEW practitioner status distinction
**Severity:** Low
**Evidence:** `fayed-mobile/src/providers/AuthProvider.tsx` lines 486-500 only checks `practitionerStatus === "APPROVED"`. Any status other than "APPROVED" (including PENDING_REVIEW, SUSPENDED, REJECTED) is redirected to `application-status`. The `application-status.tsx` screen handles these states, but the redirect logic lumps all non-APPROVED statuses together. The practitioner dashboard tabs (index, sessions, availability, more) are hidden for all non-APPROVED statuses, which is correct.
**Module:** Practitioner Auth
**Risk:** A practitioner with status SUSPENDED or REJECTED is treated the same as PENDING_REVIEW in terms of routing (redirected to application-status). The application-status screen then shows the correct state, but the routing decision doesn't distinguish between them. Functionally acceptable since all non-APPROVED practitioners are blocked from operational routes.
