### AUDIT-106: `replaceAll("_", " ")` fallback renders raw enums in messages

**Evidence:**
- `fayed-mobile/src/features/messages/utils.ts` line 146 (getParticipantStatusLabel):

- `fayed-mobile/src/features/messages/utils.ts` line 296 (getConversationStatusLabel):
  ```typescript

    return status.replaceAll("_", " ");  // English fallback renders raw enum
  }
  ```
- `fayed-mobile/src/features/messages/utils.ts` line 311 (same function default):
  ```typescript
  default:
    return status.replaceAll("_", " ");  // fallback
  ```
- `fayed-mobile/src/features/messages/utils.ts` line 350 (getMessageStatusLabel default):
  ```typescript
  default:
    return status.replaceAll("_", " ");  // fallback
  ```
- `fayed-mobile/src/features/messages/utils.ts` line 405 (getMessageSenderRoleLabel):


**Impact:** When backend returns a status enum not explicitly handled in the switch statements (e.g., a new `REASSIGNED` or `TRANSFERRED` status), the raw underscore-separated string is rendered directly in the UI. For English locale, the check `if (!isArabicLocale(locale)) { return status.replaceAll("_", " ") }` at line 296 means even common statuses fall through to the default English formatting. The Arabic locale has switch cases for OPEN, PENDING, CLOSED, etc. but English falls through for all unhandled statuses.

### AUDIT-107: `formatNotificationType` transforms `typeSlug` but is NOT i18n-aware

**Evidence:**
- `fayed-mobile/src/features/patient/profile/account-utils.ts` lines 16-22:
  ```typescript
  export function formatNotificationType(typeSlug: string) {
    return typeSlug
      .split("_")
      .filter(Boolean)
      .map((part) => part[0] + part.slice(1).toLowerCase())
      .join(" ");
  }
  ```
- `fayed-mobile/app/(patient)/profile-notifications.tsx` line 162:
  ```typescript
  {formatNotificationType(item.typeSlug)}
  ```

**Impact:** `typeSlug` values like `messages_session_message_received` are humanized to "Messages session message received" via string manipulation. No `t()` is used. There is no translation namespace for notification type slugs, so the humanized string appears in both English and Arabic without translation. Additionally, since this is client-side string manipulation rather than a lookup, any new notification types added by backend will automatically render in this format rather than being explicitly translated.

## RTL/LTR Direction Findings

### AUDIT-113: `I18nManager.isRTL` used directly in practitioner `_layout.tsx` instead of centralized helper

**Evidence:**
- `fayed-mobile/app/(practitioner)/_layout.tsx` line 27:
  ```typescript
  const isRTL = I18nManager.isRTL;
  ```

**Contrast with correct pattern** in `fayed-mobile/app/(patient)/sessions.tsx` line 51:
  ```typescript
  const direction = getAppDirection(i18n.language);
  ```

**Impact:** The practitioner tab layout reads `isRTL` from the global `I18nManager` rather than from the current i18n language context. This could desynchronize from the actual language setting, especially during hot-reload or language changes. The `getAppDirection(i18n.language)` utility in `src/i18n/direction.ts` should be used consistently.

---

### AUDIT-114: Direction utility is well-centralized - `src/i18n/direction.ts`

**Evidence:**
- `fayed-mobile/src/i18n/direction.ts` lines 5-15:
  ```typescript
  export function isCurrentLanguageRtl(language?: string | null): boolean {
    if (typeof language === "string" && language.length > 0) {
      return language.toLowerCase().startsWith("ar");
    }
    return I18nManager.isRTL;
  }

  export function getAppDirection(language?: string | null): AppDirection {
    return isCurrentLanguageRtl(language) ? "rtl" : "ltr";
  }
  ```

**Status:** This is the correct pattern. It properly checks language code first and falls back to I18nManager.

## Open Questions

1. **AUDIT-126 (Open):** Why does `en.json` have two identical `patientSessionsFlow.presentationStatus` entries (lines 792 and 2319)? This duplication should be investigated and deduplicated.

2. **AUDIT-127 (Open):** The `packagePurchases.presentationStatuses` (plural, line 1510 in en.json) uses a different key name than `presentationStatus` (singular). Is this intentional or a naming inconsistency that could cause confusion?

3. **AUDIT-128 (Open):** The practitioner sessions detail screen (`app/(practitioner)/sessions/[id].tsx`) uses `StatusBadge` with `t(\`practitioner.presentationStatus.${session.presentationStatus}\`)` at line 352, but the practitioner namespace in translations does NOT have a `presentationStatus` subtree. Was this intended to reuse `patientSessionsFlow.presentationStatus`?

4. **AUDIT-129 (Open):** The `MessagesInboxScreen.tsx` uses `getHumanStatusFallback` which applies `replaceAll("_", " ")` as a catch-all for English, but the same function has explicit Arabic translations for OPEN, PENDING, CLOSED, etc. This means English gets "OPEN" as "Open" from the switch but other statuses like "ESCALATED" would render as "ESCALATED" (raw). Should all statuses be explicitly handled?
