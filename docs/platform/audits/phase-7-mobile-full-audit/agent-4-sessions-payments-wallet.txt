## Daily/External URL Findings

### External URL Validation (AUDIT-108)
**Verdict: PASS**

Both screens use `normalizeAllowedExternalUrl` from `fayed-mobile/src/lib/external-url.ts`:

```typescript
const BASE_ALLOWED_EXTERNAL_PROTOCOLS = new Set(["https:", "fayed:"]);

export function isAllowedExternalUrl(input: string) {
  // ...
  try {
    const parsed = new URL(value);
    if (BASE_ALLOWED_EXTERNAL_PROTOCOLS.has(parsed.protocol)) {
      return true;
    }
    return typeof __DEV__ !== "undefined" && __DEV__ && parsed.protocol === "http:";
  } catch {
    return false;
  }
}
```

Patient (`fayed-mobile/app/(patient)/sessions/[id].tsx`, line 144-148):
```typescript
const safeJoinUrl = normalizeAllowedExternalUrl(joinUrl);
if (!safeJoinUrl) {
  setJoinError(t("patientSessionsFlow.detail.joinError"));
  return;
}
await Linking.openURL(safeJoinUrl);
```

Practitioner (`fayed-mobile/app/(practitioner)/sessions/[id].tsx`, line 236-241): Same safe pattern with early return on null.

**Dev-mode note**: `http:` is allowed in `__DEV__` builds only. This is expected for local development but must not be present in production builds.

### roomUrl Origin is Backend-Controlled (AUDIT-109)
**Verdict: PASS**

The `roomUrl` and `joinToken` are returned by the backend (`GET /patients/me/sessions/:id/runtime/join` and `GET /practitioners/me/sessions/:id/runtime/join`). The mobile app does not construct or guess these values — it receives them from the backend contract and passes them through without modification (except appending the `t` query param for Daily rooms).

## Payment Return Findings

### Payment Return State Handling (AUDIT-110)
**Verdict: PASS**

`fayed-mobile/app/(patient)/sessions/[id]/payment-return.tsx` handles the complete state matrix:

- `isConfirmed` → success card with checkmark (line 204-215)
- `isExpired` → expired card (line 218-229)
- `isCancelled` → cancelled card (line 232-243)
- `PENDING_PAYMENT + redirectStatus=failed` → failed card (line 246-251)
- `PENDING_PAYMENT + redirectStatus=canceled` → canceled card (line 254-265)
- `PENDING_PAYMENT + pollingActive` → verifying/syncing card (line 268-279)
- `PENDING_PAYMENT + !pollingActive` → "check again" card (line 283-294)
- Default fallback → generic pending card (line 297-307)

The `normalizePaymentRedirectStatus` function (`fayed-mobile/src/features/patient/payments/return-utils.ts`) correctly normalizes `redirect_status`, `success`, and `pending` params from multiple providers (Paymob, Stripe hosted, wallet callback).

## Wallet/Finance Findings

### Currency Formatting (AUDIT-114)
**Verdict: PASS**

Practitioner wallet (`fayed-mobile/app/(practitioner)/finance/wallet.tsx`) uses `formatMoney` from `fayed-mobile/src/features/practitioner/finance/utils.ts` (line 99-105, 113-120, etc.):

```typescript
export function formatMoney(
  amount: string | number,
  currency: string | null | undefined,
  locale: string,
  fallbackText = "-",
) {
  const currencyCode = currency?.trim().toUpperCase();
  if (!currencyCode) {
    return fallbackText;
  }
  return new Intl.NumberFormat(normalizeLocale(locale), {
    style: "currency",
    currency: currencyCode,
    maximumFractionDigits: 2,
  }).format(safeNumber(amount));
}
```

Uses native `Intl.NumberFormat` with the currency code from the backend. For EGP amounts the locale is normalized to `ar-EG` which produces EGP-appropriate formatting.

### Presence Heartbeat (AUDIT-118)
**Verdict: PASS**

`usePractitionerPresenceHeartbeat` in `fayed-mobile/src/features/practitioner/presence/hooks.ts` (lines 64-114) sends a heartbeat every 60 seconds (`HEARTBEAT_INTERVAL_MS = 60_000`) via `POST /practitioners/me/presence/heartbeat` when the app is in the `active` state. ### Availability Timezone-Aware (AUDIT-119)
**Verdict: PASS**

Practitioner availability (`fayed-mobile/app/(practitioner)/availability/index.tsx`):
- `availabilityTimeZone` (line 115-118): derived from `normalizeIanaTimeZone(availabilityData?.timezone)` — the timezone comes from the backend's availability record.
- `summaryCounts` (lines 162-203): Uses `getDatePartsInTimeZone(new Date(), availabilityTimeZone)` to compute current weekday and minute-of-day in the correct timezone. Falls back to local JS clock only if the backend timezone is unavailable (lines 176-179).
- `scheduleTimeLabel` (lines 205-214): Displays `formatTimeZoneLabel(availabilityTimeZone, { locale, includeOffset: true })` — the timezone label comes from the backend, formatted with the correct offset.
- Weekly slots are saved with `timezone: availabilityData.timezone` (line 278), sent to `PUT /practitioners/me/availability` — preserving the backend's timezone, not the device's local timezone.

### No Hardcoded Timezone City Names
**Verdict: PASS**

The `_formatTimezoneLabel` function (lines 700-729 in the practitioner session detail screen) contains hardcoded city maps (Cairo, Riyadh, Dubai, Kuwait, Doha), but this function is defined and never called — it is dead code. The actual timezone display on the availability screen uses `formatTimeZoneLabel` from `time-formatting.ts`, which uses the IANA timezone name directly.

### Availability Summary Counts Timezone Awareness (AUDIT-120)
**Verdict: PASS with minor fallback concern**

`fayed-mobile/app/(practitioner)/availability/index.tsx`, lines 162-203: When computing `currentWeekday` and `currentMinuteOfDay` for the availability summary chips (today/upcoming/active/completed counts), the code uses `getDatePartsInTimeZone(new Date(), availabilityTimeZone)` when `availabilityTimeZone` is available. If `availabilityTimeZone` is null (no backend data), it falls back to `new Date().getDay()` and `new Date().getHours() * 60 + new Date().getMinutes()` — which uses the device's local timezone. This is a minor edge case; the fallback only affects the visual counts if the backend timezone is unavailable.

## Open Questions

1. **Dev-mode http: allowance**: `normalizeAllowedExternalUrl` allows `http:` in `__DEV__` builds. Is there a production build flag that also enables this? If so, this should be audited separately as a security concern. Evidence: `fayed-mobile/src/lib/external-url.ts` line 15.

2. **Wallet-only payment flow**: In `pay.tsx` (lines 741-750), when `payment.status === "CAPTURED" || payment.status === "AUTHORIZED"` (wallet-only, no redirect), the screen navigates to `payment-return` with `redirect_status: succeeded`. Is the 45-second polling timeout (line 29: `MAX_POLL_DURATION_MS = 45_000`) sufficient for wallet-only payments, or could it resolve faster and show a false "pending" state?

3. **Join URL token encoding**: The Daily join URL appends `?t=<token>` or `&t=<token>`. Is there a maximum URL length consideration for extremely long room URLs? This is low risk but worth confirming with the backend team.

4. **`READY_TO_JOIN` status in confirmed statuses set**: `payment-return.tsx` line 23 includes `READY_TO_JOIN` in `CONFIRMED_SESSION_STATUSES`. This is used to stop polling. If a session is `READY_TO_JOIN` but the payment webhook is delayed, could the screen redirect prematurely before payment confirmation?

5. **Availability exception timezone**: When creating an exception via `handleCreateOrUpdateException` (line 331), the `startsAt`/`endsAt` are parsed from local datetime input (via `parseLocalDateTimeToIso`) and sent as ISO strings. The backend stores them and `formatAvailabilityExceptionRange` (line 1231) formats them back using the availability's timezone. Is the input assumed to be in the availability timezone or the device timezone?
