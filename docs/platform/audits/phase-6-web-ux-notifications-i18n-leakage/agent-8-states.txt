1. LOADING STATES
=================================================================

[P2] FINDING: ZERO route-level loading.tsx files exist anywhere in the application.

  Confirmed absent from ALL critical route groups:
  - fayed-frontend-v1/src/app/[locale]/(patient)/patient/ -- NO loading.tsx
  - fayed-frontend-v1/src/app/[locale]/(practitioner)/practitioner/ -- NO loading.tsx
  - fayed-frontend-v1/src/app/[locale]/(admin)/admin/ -- NO loading.tsx
  - fayed-frontend-v1/src/app/[locale]/(public)/ -- NO loading.tsx (only error.tsx at practitioners/)

  Only two loading.tsx-equivalent state files exist in the entire app:
  - fayed-frontend-v1/src/app/[locale]/error.tsx (this is a Next.js Error Boundary, not a streaming loader)
  - fayed-frontend-v1/src/app/[locale]/(public)/practitioners/error.tsx (Error Boundary)

  The app relies entirely on component-level isLoading checks (React Query hooks).
  Next.js App Router streaming/suspense is effectively disabled for all 80+ routes.
  No fallback UI is shown during initial route navigation -- users see a white screen
  or the browser's default loading indicator.

  Evidence: find output confirms zero loading.tsx files under (patient), (practitioner), (admin).

3. ERROR STATES
=================================================================

[P2] FINDING: AppErrorFallback logs full stack traces to console in development.

  File: fayed-frontend-v1/src/components/shared/AppErrorFallback.tsx:14-21

  In development mode:
    if (process.env.NODE_ENV === "development") {
      console.error("[app] route error boundary:", {
        name: error.name,
        message: error.message,
        stack: error.stack,    <-- FULL STACK TRACE
      });
    }

  This means any patient-facing error that gets caught by the error boundary
  (including auth failures, validation errors, network errors) will have its
  full stack exposed in browser DevTools. While this is behind a development
  guard, patient-facing errors on staging could leak implementation details.

  Evidence: lines 14-21.

---

[P2] FINDING: AppErrorFallback does not distinguish error types in copy.

  File: fayed-frontend-v1/src/components/shared/AppErrorFallback.tsx:57-73

  All error types (UNAUTHORIZED, FORBIDDEN, NOT_FOUND, SERVICE_UNAVAILABLE, etc.)
  render the same generic title and description:
    title: t("errors.fallback.title")     <-- same for all types
    description: t("errors.fallback.description")  <-- same for all types

  A 403 Forbidden and a 500 Server Error show identical UI.
  The ERROR_TYPE_STYLES at lines 31-39 color-code the error type badge,
  but the main heading and description are identical across all error types.

  Evidence: lines 57-73, 31-39.

=================================================================

[P2] FINDING: AdminPermissionGate shows a raw "Loading..." text during permission check.

  File: fayed-frontend-v1/src/components/admin/AdminPermissionGate.tsx:59-65

  When isLoading is true:
    return (
      <div className="flex min-h-[60vh] items-center justify-center px-4 text-sm text-text-muted">

      </div>
    );

  No skeleton component, no spinner, no "Checking permissions..." label.
  This is the loading state for an authorization check -- should be more explicit.

  Evidence: lines 59-65.

---

[P2] FINDING: AdminForbiddenView exists and is properly designed for admin routes.
  File: fayed-frontend-v1/src/components/admin/AdminForbiddenView.tsx:17-54

  The AdminForbiddenView is well-designed:
  - Uses an SVG shield icon (not emoji)
  - Title: "Access Denied" (clear and calm)
  - Copy: "Contact your administrator if you believe this is a mistake."
  - CTA: "Go to Dashboard" link
  - No mention of which permission is missing
  - No stack trace or technical details

  This is the GOLD STANDARD for permission-denied UX in this codebase.
  However, it is ONLY used inside AdminPermissionGate for admin routes.

  Evidence: AdminPermissionGate.tsx lines 68, 76; AdminForbiddenView.tsx full file.

---

[P2] FINDING: Patient and practitioner routes have NO dedicated permission-denied component.

  There is no PatientPermissionGate or PractitionerPermissionGate equivalent.
  If a patient navigates to a route they don't have access to, the error
  propagates to the nearest error boundary (AppErrorFallback), which shows
  the generic "Something went wrong" page -- identical to a 500 error.

  Evidence: No files matching PatientPermissionGate or PractitionerPermissionGate
  exist in the codebase.

6. CLINICAL WARMTH CHECK
=================================================================

[PASS] PatientSessionsPanel error copy (lines 90-92, 147-149) uses:
  "Could not load sessions" / "Please try refreshing the page."
  Does NOT blame the user ("You did something wrong") or expose stack traces.

[PASS] Patient session cancellation modal (lines 759-763) shows:
  "isCancellable ? t("detail.cancelConfirm.policyNote") : t("detail.cancelConfirm.cannotProceed")"
  The cannotProceed message is calm: "Cancellation is not available for this session."
  Does not say "You cannot cancel."

[PASS] AppErrorFallback (lines 65-73) uses:
  title: "Something went wrong" (not "Error!" or "Failure!")
  description: calm explanatory text, not blaming the user.

[FAIL] AppErrorFallback (lines 14-21) -- console.error with full stack in dev mode
  could expose implementation errors to anyone with browser access in dev/staging.

[OBSERVATION] The StateCard component (ContentStates.tsx lines 19-55) does not
  have a variant that shows a warning/info tone -- it only handles error/empty states.
  Error states that are non-critical (e.g., "could not load optional data") use the
  same card style as critical errors.
