================================================================================
PHASE 6 WEB UX I18N LEAKAGE AUDIT — FAYED FRONTEND V1
Auditor: Agent 6 (Claude Code)
Date: 2026-06-17
Scope: fayed-frontend-v1 (all *.tsx files, messages/*.json)
================================================================================

SUMMARY
-------
Seven distinct leakage patterns found across the frontend. Raw backend enums
are rendered directly to users via replaceAll("_", " ") in session chat and
workspace components. i18n key namespaces are well-structured overall, but two
presentationStatus values (IN_PROGRESS, JOINABLE) are missing from the
presentationStatus namespace in both locale files. PermissionKey values appear
in visible admin UI labels on the permissions screen.

================================================================================
PATTERN 1 — RAW ENUM RENDERED VIA replaceAll("_", " ") IN VISIBLE JSX
================================================================================

FINDING P-1-1: presentationStatus raw enum displayed in session chat badge
File:  fayed-frontend-v1/src/features/chat/components/SessionChatPanel.tsx
Line:  355
Code:
  {session?.presentationStatus.replaceAll("_", " ")}

The raw backend enum value (e.g. "IN_PROGRESS", "JOINABLE", "NO_SHOW") has its
underscores replaced with spaces and is rendered directly as a visible badge
text. This bypasses the i18n layer entirely. Users see "IN PROGRESS" instead
of the translated "In progress" / "قيد التنفيذ".

---

Same pattern — raw enum is string-manipulated and displayed as a visible label
in the session lane list. The key "statusLabel" is passed to a list item

---

FINDING P-1-3: Raw enum displayed via replace(/_/g, " ") in admin moderation report
File:  fayed-frontend-v1/src/features/admin/moderation-reports/components/AdminModerationReportDetailScreen.tsx
Line:  59

This is used in formatSnapshotLabel() which formats moderation report field

PATTERN 2 — MISSING presentationStatus i18n KEYS
================================================================================

FINDING P-2-1: presentationStatus namespace missing IN_PROGRESS and JOINABLE
Files:
  - fayed-frontend-v1/messages/en/sessions.json (lines 66-76)
  - fayed-frontend-v1/messages/ar/sessions.json (lines 66-76)

presentationStatus namespace content (EN):
  UPCOMING, JOINABLE, IN_PROGRESS, COMPLETED, CANCELLED, ENDED, UNAVAILABLE,
  NO_SHOW, UNDER_REVIEW

Actual keys present in presentationStatus object:
  UPCOMING         ✓
  JOINABLE         ✗ MISSING
  IN_PROGRESS      ✗ MISSING
  COMPLETED        ✓
  CANCELLED        ✓
  ENDED            ✓
  UNAVAILABLE      ✓
  NO_SHOW          ✓
  UNDER_REVIEW     ✓

The two active/live statuses (JOINABLE, IN_PROGRESS) have NO i18n entry in the
presentationStatus namespace. Components that use t(`detail.presentation.${session.presentationStatus}.note`)
will fail to resolve JOINABLE and IN_PROGRESS keys, falling back to the raw key
or rendering nothing. Meanwhile the list.presentationHints namespace DOES have
JOINABLE and IN_PROGRESS entries, creating an inconsistency.

---

FINDING P-2-2: presentationHints IN_PROGRESS key present but inconsistent
File:  fayed-frontend-v1/messages/en/sessions.json (line 34)
File:  fayed-frontend-v1/messages/ar/sessions.json (line 34)

list.presentationHints.IN_PROGRESS = "This session is live now." (EN)
list.presentationHints.JOINABLE    = "You can enter this session now." (EN)

But there is no matching presentationStatus.IN_PROGRESS or
presentationStatus.JOINABLE entry. Components that use the presentationStatus
namespace for these values will not find translations.

PATTERN 7 — toLowerCase() ON ENUM-LIKE VALUES IN VISIBLE CONTEXT
================================================================================

FINDING P-7-1: toLowerCase() on currency codes in patient payments history
File:  fayed-frontend-v1/src/features/payments/components/PatientPaymentsHistoryPanel.tsx
Lines:  340, 342, 381
Code (line 381):
  label={t(`history.summary.${item.currencyCode.toLowerCase()}` as Parameters<typeof t>[0])}

Currency codes (e.g. "USD", "EUR") are lowercased and used as i18n keys. If
the i18n file uses uppercase keys (e.g. "USD" not "usd"), this will fail to
resolve. The fallback behavior depends on the i18n library configuration.

---

FINDING P-7-2: toLowerCase() on run/scope/trigger values in accounting reconciliation
File:  fayed-frontend-v1/src/features/admin/accounting-reconciliation/components/FinancialReconciliationScreen.tsx

Extensive use of toLowerCase() on backend enum values (run.status, run.scope,
run.trigger, issue.severity, issue.status) to construct i18n keys. If the i18n
files use uppercase or camelCase keys, all these lookups will fail silently.
Example (line 539):
