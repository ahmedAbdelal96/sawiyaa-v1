FINDING 1 — RAW ENUM LEAK IN CHAT EMBEDDED HEADER
Severity: HIGH
File: fayed-frontend-v1/src/features/chat/components/SessionChatPanel.tsx
Lines: 354-356

Code:
  <span className="rounded-full bg-teal-50/70 border border-teal-100/30 px-2.5 py-0.5 text-[10px] font-bold text-teal-700 dark:bg-teal-950/40 dark:text-teal-400">
    {session?.presentationStatus.replaceAll("_", " ")}
  </span>

Evidence: presentationStatus is rendered raw with replaceAll("_", " ") — it
produces strings like "UNDER REVIEW", "READY TO JOIN", "NO SHOW". This is
patient-visible in the chat header. It bypasses i18n entirely. The correct
pattern is to use the t() function with the sessions namespace, e.g.:
  t(`presentationStatus.${session.presentationStatus}` as Parameters<typeof t>[0])

No other patient-facing component was found to render presentationStatus raw.
This is the sole instance.

Impact: Patient sees raw SCREAMING_SNAKE_CASE in the chat UI. Also a
presentationStatus value of UNDER_REVIEW would produce "UNDER REVIEW" which
carries a clinical/stigmatizing connotation when not properly localized.

FINDING 2 — MISSING i18n LABEL FOR UNDER_REVIEW IN detail.presentation
Severity: MEDIUM
Files:
  fayed-frontend-v1/messages/en/sessions.json — line 66-76 (presentationStatus)
  fayed-frontend-v1/messages/ar/sessions.json — line 66-76

Evidence: The presentationStatus section in both locale files defines
UNDER_REVIEW:
  "presentationStatus": {
    "UNDER_REVIEW": "Under review"   -- en
    "UNDER_REVIEW": "قيد المراجعة"  -- ar
  }

However, the detail.presentation block (en lines 91-131, ar lines 91-132)
has NO entry for UNDER_REVIEW. It only covers:
  UPCOMING, JOINABLE, IN_PROGRESS, COMPLETED, CANCELLED, ENDED, UNAVAILABLE, NO_SHOW

In PatientSessionDetailPanel.tsx lines 267-275, the code calls:
  t(`detail.presentation.${session.presentationStatus}.title` as Parameters<typeof t>[0])
  t(`detail.presentation.${session.presentationStatus}.note` as Parameters<typeof t>[0])
  t(`detail.presentation.${session.presentationStatus}.closeout` as Parameters<typeof t>[0])

If presentationStatus is UNDER_REVIEW, these t() calls will return the key
itself as a fallback string (next-intl returns "[detail.presentation.UNDER_REVIEW.title]"
on missing keys), producing raw key leakage in the session detail runtime section.

Impact: If a session enters UNDER_REVIEW state, the detail page shows raw
i18n key strings instead of human-readable labels. This is a fragile-state leak.

FINDING 3 — NO loading.tsx / error.tsx IN PATIENT ROUTES
Severity: MEDIUM
Path: fayed-frontend-v1/src/app/[locale]/(patient)/patient/*

Evidence: Glob search for **/patient*/**/loading.tsx and **/patient*/**/error.tsx
returned no matches. The patient route tree has 35+ page routes including:
  patient/sessions/page.tsx
  patient/sessions/[id]/page.tsx
  patient/sessions/[id]/chat/page.tsx
  patient/sessions/[id]/pay/page.tsx
  patient/care-chat/page.tsx
  patient/care-chat/[id]/page.tsx
  patient/payments/page.tsx
  patient/wallet/page.tsx
  patient/messages/page.tsx

None of these have Next.js loading.tsx or error.tsx boundary files. The
components handle loading/error internally (via ListStateSkeleton and
StateCard), which is acceptable for component-level loading states. However,
route-level error.tsx boundaries are the standard Next.js pattern for catching
Unexpected Runtime Errors in the segment.

Impact: An unexpected error in a patient session page would show the default
Next.js error overlay or a blank page rather than a coherent, branded error
state, if no error boundary is provided by the parent layout.

FINDING 13 — MISSING UNDER_REVIEW IN detail.presentation CLINICAL WARMTH NOTE
Severity: LOW (Contextual)
Files:
  fayed-frontend-v1/messages/en/sessions.json
  fayed-frontend-v1/messages/ar/sessions.json

Note: Even if the UNDER_REVIEW i18n gap (Finding 2) is fixed by adding
title/note/closeout entries, the tone for UNDER_REVIEW is "warning" in
SessionStatusBadge.tsx line 26. A "warning" amber tone for a clinical
status that may reflect a session under clinical review could feel
stigmatizing to a patient. The clinical warmth principle would suggest
a more neutral or even informational tone for this state, not warning
(which is reserved for actionable patient decisions).

This is a design-level concern, not a bug.
