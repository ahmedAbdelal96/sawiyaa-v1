FINDING 1: AUDIT-059 CONFIRMED -- UNREAD BADGE COUNTS IN_APP ONLY
------------------------------------------------------------------------
Area: Unread Badge / Notification Count
Severity: FATAL
Confirmed in:
  - user-notifications.api.ts:20-25 -- getMyUnreadNotificationCount()
  - user-notifications.types.ts:62-65 -- UserUnreadNotificationCountResponseData
  - UserNotificationDropdown.tsx:47-48

Evidence:
  The useMyUnreadNotificationCount() hook hits "/notifications/me/unread-count"
  which is a separate endpoint from the messages shell's unread summary.  The
  API response shape is:

    { item: { unreadCount: number } }

  This is a single integer with no channel breakdown.  Unlike the messages
  shell (use-unified-unread-badge.ts) which fetches
  /chat/conversations/unread-summary and gets per-lane message counts, the
  notification unread count endpoint has no such breakdown.

  The UserNotificationDropdown.tsx line 48:
    const unreadCount = unreadCountQuery.data?.item.unreadCount ?? 0;

  This value is displayed in the bell badge (line 83-87).  If the backend
  counts only IN_APP notifications here while PUSH/EMAIL/SMS notifications
  carry unread state, the badge will show an incorrect (lower) count.

  NOTE: Phase 5 AUDIT-059 was issued for mobile native.  This confirms the
  same architectural pattern exists on the web frontend (separate endpoint,
  no channel breakdown visible in the response).

FINDING 2: ADMIN-070 -- bodySnapshot Rendered as innerHTML (XSS Risk)
------------------------------------------------------------------------
Area: Admin Notification Detail / HTML Rendering
Severity: FATAL
Confirmed in:
  AdminNotificationDetailsPanel.tsx:418-426

Evidence:
  The bodySnapshot field is rendered inside a <p> tag using dangerouslySetInnerHTML
  is NOT used -- however, the text is rendered directly as React children
  (whitespace preserved), meaning the string is not escaped:

    {item.bodySnapshot && (
      <div className="rounded-xl bg-surface-secondary/70 p-3.5 ...">
        <span className="text-[10px] text-text-muted block font-bold mb-1">
          {isAr ? "محتوى الرسالة" : "Message Body"}
        </span>
        <p className="text-xs text-text-primary dark:text-white/80
                      whitespace-pre-wrap leading-relaxed">
          {sanitizeNotificationDisplayText(item.bodySnapshot, locale, t)}
        </p>
      </div>
    )}

  The sanitizeNotificationDisplayText() function (line 32) does strip/redact
  full-URL text blocks, but it is a TEXT transformation function -- it does NOT
  escape HTML entities.  If item.bodySnapshot contains e.g.
  "<script>evil()</script>" it would be rendered as HTML by the browser.

  The same applies to titleSnapshot (line 399-406) and subjectSnapshot
  (line 408-416).

  Additionally, the "copy JSON" diagnostic feature (line 120-129) does redact
  URLs before copying to clipboard, but this does not protect the rendered UI.

CROSS-CUTTING OBSERVATIONS
------------------------------------------------------------------------

1. UNIFIED MESSAGING SHELL UNREAD vs NOTIFICATION UNREAD
   The messages shell uses a SEPARATE API endpoint
   (/chat/conversations/unread-summary) to compute its unread count
   (use-unified-unread-badge.ts:37).  This returns per-lane unread message
   counts.  The notification system's unread count is a SEPARATE query hitting
   /notifications/me/unread-count.  These two counts may diverge over time,
   potentially showing different badge values on the bell icon vs. inside the
   messages shell.  This is an architectural observation worth monitoring.

2. ADMIN TECHNICAL DETAILS PANEL -- FULLY ADMIN-ONLY
   The technical details card (AdminNotificationDetailsPanel.tsx:439-522)
   is only visible to admins.  Findings 2 and 3 (HTML rendering of
   bodySnapshot, userId in technical panel) are admin-only attack surfaces.

3. NO NOTIFICATION PAGE FOR PATIENT/PRACTITIONER
   There is no deep-linkable notification center for patient or practitioner
   roles.  All notification interaction happens via the dropdown.  If a
   notification needs to be shared or bookmarked, there is no URL to use.
