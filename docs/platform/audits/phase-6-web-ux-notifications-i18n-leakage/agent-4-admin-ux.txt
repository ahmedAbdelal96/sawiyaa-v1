FINDING 1: MISSING PERMISSION GATE ON care-chat/[id] — AUDIT-068 REGRESSION
================================================================================
Severity: HIGH
Status: UNMITIGATED (previously flagged in Phase 5)

File: fayed-frontend-v1/src/app/[locale]/(admin)/admin/care-chat/[id]/page.tsx
Lines: 1-22 (entire file)

Finding:
  The route /admin/care-chat/[id] has NO AdminPermissionGate wrapping its
  content. The parent /admin/care-chat page (line 24-32) correctly gates
  access with requiredPermissions for CARE_CHAT_REQUEST_READ_ADMIN,
  CARE_CHAT_CONVERSATION_READ_ADMIN, and CARE_CHAT_REQUEST_DECIDE, but the
  detail page [id] passes through directly to AdminCareChatRequestScreen
  without any permission check.

Evidence:
  ------
  18: export default async function AdminCareChatRequestPage({ params }: Props) {
  19:   const { locale, id } = await params;
  20:   setRequestLocale(locale);
  21:   return <AdminCareChatRequestScreen requestId={id} />;
  22: }
  ------

Compare with parent route (care-chat/page.tsx, lines 24-32):
  <AdminPermissionGate
    requiredPermissions={[
      PermissionKey.CARE_CHAT_REQUEST_READ_ADMIN,
      PermissionKey.CARE_CHAT_CONVERSATION_READ_ADMIN,
      PermissionKey.CARE_CHAT_REQUEST_DECIDE,
    ]}
  >

Impact:
  Any admin-class user who can guess or navigate to a care-chat request ID
  can view its detail page, bypassing the permission check present on the
  list view.

Recommendation:
  Wrap AdminCareChatRequestScreen with AdminPermissionGate using the same
  requiredPermissions array as the list page.

FINDING 4: providerRoomRef DIAGNOSTIC IDENTIFIER EXPOSURE IN ATTENDANCE TIMELINE
================================================================================
Severity: MEDIUM
Status: NEW FINDING

File: fayed-frontend-v1/src/features/admin/session-runtime/components/AdminSessionAttendanceSection.tsx
Lines: 119-124

File: fayed-frontend-v1/src/features/admin/session-runtime/components/AdminSessionInspectorTimeline.tsx
Lines: 106-114

Finding:
  The providerRoomRef field is rendered in the expandable "providerDetails"
  section of each timeline event. This is a provider-specific room identifier
  that, depending on the provider implementation, could function as a join
  token, room URL component, or session credential.

Evidence (AdminSessionAttendanceSection.tsx):
  ------
  119:           <p className="break-all">
  120:             <span className="font-semibold text-text-primary dark:text-white/90">
  121:               {t("attendance.timeline.fields.providerRoomRef")}
  122:             </span>
  123:             {item.providerRoomRef ?? "-"}
  124:           </p>
  ------

Evidence (AdminSessionInspectorTimeline.tsx, lines 106-114):
  ------
  106:             {event.providerRoomRef ? (
  107:               <div className="flex items-center justify-between gap-3 border-b border-border-light py-1.5 dark:border-white/8">
  108:                 <span className="text-[11px] text-text-muted">
  109:                   {providerRoomRefLabel}
  110:                 </span>
  111:                 <span className="font-mono text-[11px] text-text-secondary break-all">
  112:                   {event.providerRoomRef}
  113:                 </span>
  114:               </div>
  115:             ) : null}
  ------

Note: AdminSessionInspectorRawEvidence.tsx (line 21) DOES sanitize this field
in its raw JSON export — it includes providerRoomRef in the sanitized payload
shown to admins, which is appropriate for diagnostic purposes but confirms
the field is considered sensitive.

Impact:
  Admins viewing session attendance timeline events can see raw provider room
  references. If these are session join tokens or contain embedded secrets,
  this could enable unauthorized session access.

Recommendation:
  Treat providerRoomRef as a session credential. Consider:
  1. Masking the value (showing only last 4 characters) in the timeline UI
  2. Requiring an elevated permission (e.g., SESSIONS_MANUAL_DECISIONS_WRITE)
     to reveal the full value
  3. Confirming with backend whether providerRoomRef is a secrets-bearing
     token or merely an opaque room identifier

FINDING 7: RAW PERMISSION KEYS IN ADMIN PERMISSION CATALOG (LOW RISK)
================================================================================
Severity: INFO
Status: ACCEPTABLE — internal diagnostic/tooling use

File: fayed-frontend-v1/src/features/admin/users/constants/admin-permission-catalog.ts
Lines: 18-345 (entire file)

Finding:
  The file admin-permission-catalog.ts maps every PermissionKey enum value
  to a labelKey, descriptionKey, module, and risk rating. This is a catalog
  used for the admin user permissions management UI (used by
  AdminUserPermissionsScreen to render the permission matrix).

These keys are used internally by the permissions management screen and do
not appear to be rendered directly as raw strings in user-facing labels.
The PermissionKey values are used as `key` fields in the catalog items,
which are then used to look up i18n labels via labelKey/descriptionKey.

The raw PermissionKey strings (e.g., "admin-users.read") do NOT appear in
any user-facing UI — they are only in the internal catalog data structure.

This is acceptable because:
1. The catalog is only loaded for the permissions management screen
2. i18n label keys are used for all displayed text
3. Permission keys are a documented internal API contract, not secrets

FINDING 9: DESTRUCTIVE ACTION CONFIRMATION DIALOGS
================================================================================
Severity: LOW (GOOD)
Status: PRESENT AND ADEQUATE

File: fayed-frontend-v1/src/features/admin/chat-conversations/components/AdminChatConversationDetailScreen.tsx
Lines: 849-935

File: fayed-frontend-v1/src/features/admin/settlements/components/AdminSettlementDetailScreen.tsx
Lines: 577-685

Finding:
  Chat conversation disable/enable actions use FormModal confirmation dialogs
  (lines 849-935 in AdminChatConversationDetailScreen), with a warning callout
  about the destructive nature of disabling.

  Settlement mark-paid and mark-failed actions in AdminSettlementDetailScreen
  use form submissions with descriptive text explaining consequences (lines
  577-685). They do NOT use modal confirmations, but they do have:
  - Disabled state when action is not available (actionState.canMarkPaid)
  - Explanatory text about what "mark paid" means (line 637)
  - Explanatory text about what "mark failed" means (line 672)
  - Descriptive field labels

  The settlement destructive actions are lower risk than chat disable because:
  1. They are gated by role check (canOperate = ADMIN or SUPER_ADMIN, line 184)
  2. They affect financial records rather than communication
  3. They produce a feedback message (success/error) after API response

Overall: Adequate. Consider adding explicit modal confirmations for
mark-paid and mark-failed if they are considered high-risk financial ops.
